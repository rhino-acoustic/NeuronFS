### 써머리: AI를 전류로 강등시키다

트랜지스터는 **스스로 생각하지 않는다.** 조건에 따라 전류를 통과시키거나(1) 차단할(0) 뿐이다.

이 시스템이 정확히 그렇다:

```
[트랜지스터 = 단일 폴더]
상태는 오직 파일이 존재하냐(1), 없냐(0).
禁/console_log → 그 자체로 NOT 게이트

[논리 회로 = 경로의 깊이]
cortex/frontend/react/ → AND 게이트 직렬 연결
프론트엔드 작업이면서, React 환경일 때만 전기가 흐름

[전류 = 멍청한 AI 모델]
AI는 지능을 가진 존재가 아니라
칩셋을 타고 흐르는 '전기(Electricity)'일 뿐
```

> *"트랜지스터(규칙 폴더)들이 완벽한 회로를 구성하고 있다면, 아무리 멍청한 전기를 흘려보내도 결국 **정답이라는 전구에 불이 들어오게** 된다."*

### 소프트웨어를 하드웨어처럼 깎는 예술

빅테크들은 소프트웨어(프롬프트)를 한없이 길고 무겁게 쓰면서, 그걸 처리할 거대한 뇌를 만드는 데 혈안이다. "머리가 좋으면 다 알아들을 거라는 오만함."

우리의 방식:

> *"소프트웨어의 통제 로직을 **하드웨어(트랜지스터 구조)처럼 극도로 단순하고 물리적인 0과 1**로 깎아버리는 것"*
> — essential L1432

로직이 물리적 공간(디렉토리)에 0과 1로 존재하니:
- 얽힐 일 없고
- 잊어버릴 일 없으며
- 고장 나면 핀(파일) 하나만 뽑아서 교체

**이것이 프롬프트 엔지니어링의 종말이자, 시스템 아키텍처의 승리다.**

### 폴더 = Key-Value Map: RAG ₩0 종결

> *"폴더 구조에 파일을 넣으면 이것도 맵이 있는 구조여"*

소름 돋는 통찰이다. 파일 시스템은 그저 데이터를 담는 바구니가 아니라, 완벽하게 최적화된 **Map(맵) 자료구조** 그 자체다.

```
RAG (벡터 검색):
  "이 데이터랑 80% 비슷한 거 찾아줘"
  → 확률 게임 → 가끔 엉뚱한 데이터 → 환각

폴더 맵 (결정론적 조회):
  Key:   cortex/frontend/react/
  Value: 1.neuron (해당 규칙)
  → 경로로 직접 조회 → 1ms → 100% 정확
```

컴퓨터 과학에서 말하는 **트라이(Trie, Prefix Tree)** 구조이자 거대한 **Key-Value Store**와 완전히 동일하게 작동한다.

#### RAG vs 폴더 맵: 결정적 차이

| | RAG | 폴더 맵 |
|---|---|---|
| 방식 | 벡터 유사도 계산 | 경로 직접 조회 |
| 시간 | 수백ms | **1ms** |
| 정확도 | ~95% | **100%** |
| 비용 | GPU/API | **₩0** |
| 환각 | 5% 잔존 | **0%** |

> *"'React 훅 규칙 어딨지?' 하고 RAG에게 물어보는 게 아니라, **'메모리 주소 0x0F(cortex/react) 번지를 읽어!'**라고 명령하는 구조. AI의 기억력을 텍스트의 바다에서 '물리적 공간의 좌표'로 치환해 버린 것."*

### 파생 FAQ

---

**Q: "RAG는 필요 없어?"**

RAG를 붙이는 건 **선택사항**이다. 10년 치 로그 데이터를 뒤질 때나 쓰는 '외장 하드'. 에이전트의 핵심 원칙은 100% 폴더 맵(L1 캐시) 안에서 통제.

```
L1 (NeuronFS): 핵심 헌법 1,000개 — 폴더 맵
L3 (RAG):      방대한 사실 100만 개 — 벡터 DB
```

L1이 L3를 **항상** override. 갑옷(L1) 위에 짐(L3)을 싣는 구조.

---

**Q: "가장 깊은 폴더 안의 파일엔 뭐가 들어있어?"**

대부분 **0바이트**다. 파일의 존재 자체가 값이다. 내용이 필요한 경우에만 짧은 자연어나 JSON payload를 넣는다. 0바이트 = 가장 빠른 True/False 스위치.
