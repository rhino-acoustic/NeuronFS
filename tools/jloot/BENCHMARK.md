## 생성 벤치마크

| 방식 | 시간 | 비고 |
|------|------|------|
| 한글 폴더 (순차) | 8m 20s | 1회성 |
| 한글 폴더 (병렬 66 goroutines) | 5m 21s | 36% 개선 |
| 난수 폴더 (순차) | 10m 01s | NTFS mkdir 자체가 병목 |

> **결론**: 생성은 느리지만 1회성. 인메모리 마운트 34ms + 조회 51ns가 핵심.

## 커맨드 사용법

```bash
# 패킹
jloot pack <dir>                    → output.jloot

# 언패킹
jloot unpack <file.jloot> -o <dir>  → 복원

# 정보
jloot info <file.jloot>

# 인메모리 검색
jloot search <file.jloot> <keyword>

# HTTP API 서버 (인메모리 마운트)
jloot mount <file.jloot> -p 7700
# → GET /get?path=43_요한복음/003장/016.md
# → GET /search?q=사랑
# → GET /ban?dir=출애굽기/020장/015절
# → GET /stats
# → POST /materialize?dir=output (백그라운드 난수폴더 생성)
```

## Jloot vs 벡터/RAG (솔직한 비교)

| | Jloot (TF-IDF) | 벡터 임베딩 |
|---|---|---|
| **원리** | 키워드 매칭 | 의미 유사도 (숫자 비교) |
| **강점** | 정확한 키워드 → 즉시 반환 | 추상적 질문 → 의미적 매칭 |
| **약점** | 추상적 질문에 약함 | 인프라/비용 필요 |
| **속도** | **0ms** | 50~500ms |
| **비용** | 0원 | 임베딩 모델 필요 |
| **거버넌스** | ✅ 禁/必 | ❌ |
| **오프라인** | ✅ | 로컬 임베딩 모델 필요 |

> **결론: 대체가 아니라 레이어.**
> - 1단계: Jloot TF-IDF (0ms, 키워드 매칭)
> - 2단계: 벡터 보완 (키워드 실패 시 의미 검색)
> - 3단계: LLM 해석 (구절 기반 답변 생성)
> - + NeuronFS 하네스 (전 과정에 거버넌스 적용)

## 하네스 환각 방어 실증 (A~F 6방식)

> 테스트: 2026-04-05 23:30 KST
> 같은 복합 질문, 같은 모델(Opus 4.6), 하네스/컨텍스트만 변화

| 방식 | 시간 | 글자수 | 환각 방어 |
|------|------|--------|:---:|
| A: bare Opus (아무것도 없이) | 76초 | 8,869 | ❌ 50+ 구절 자유 인용 |
| B: TF-IDF + 성경하네스 | 68초 | 8,950 | ⚠️ |
| C: 임베딩 + 성경하네스 | 65초 | 8,751 | ⚠️ |
| D: brain_v4 규칙 + TF-IDF | 44초 | 5,802 | ✅ |
| **E: NeuronFS 하네스 + TF-IDF** | **29초** | **3,891** | **✅ 7개 중 6개 "해당 구절 없음"** |
| F: NeuronFS 하네스 + 임베딩 | 56초 | 6,803 | ✅ 7개 중 4개 답변 가능 |

> **핵심:** bare Opus는 50+ 구절을 자유 인용하지만 검증 불가.
> NeuronFS 하네스 적용 시 Opus조차 "증거 없으면 답 안 함" → 환각 0.
> 임베딩은 TF-IDF보다 3배 더 많은 관련 구절을 찾아줌 → 답변 범위 확대.

## 포지션

> **Jloot = 로컬 AI 거버넌스 + 환각 방어 인프라**

- 군부대, 인트라넷 기업, 의료기관, 법무법인, 공장 OT망
- USB 하나로 배포 가능한 AI 거버넌스
- 클라우드 의존 제로, 비용 제로
- 비개발자가 폴더를 만들어 AI를 통제
- NeuronFS 하네스 = "모르면 모른다고 하는 능력" 강제
- 벡터 임베딩과 조합 시 추상적 질문까지 커버
