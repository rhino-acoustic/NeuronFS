# ⚡ Jloot

`34ms 마운트` · `51ns 조회` · `1.4MB 패키지` · `DB 없음` · `서버 없음` · `인터넷 없음`

## 왜 Jloot인가

벡터DB는 5MB 텍스트를 200MB RAM으로 뻥튀기하고, 쿼리당 100~500ms를 소비합니다.

Jloot은 같은 텍스트를 **1.4MB로 압축**하고, **34ms에 RAM 마운트**하여, **51나노초**에 조회합니다.

```
기존:  텍스트 → 임베딩(50ms) → 벡터DB(100ms) → 리랭킹(50ms) → 답변
Jloot: .jloot → RAM(34ms, 1회) → map[경로](51ns) → 답변
```

| 패키지 크기 | **1.4MB** | DB 서버 필요 |
| 월 비용 | **0원** | $70+ (Pinecone) |
| 오프라인 | ✅ | ❌ |
| 거버넌스 (禁/必) | ✅ | ❌ |

## 아키텍처

```
.jloot (암호화된 단일 파일)
    ↓ Mount() — 34ms
    ↓
[RAM 해시맵]
    ├── 경로 인덱스     → 51ns O(1) 조회
    ├── 거버넌스 (禁/必) → AI 차단/강제
    └── 벡터 임베딩     → 코사인 유사도 (예정)
    
    ↓ HTTP API
    ↓
로컬 AI (Ollama 등) / 클라우드 AI
```

### L1 + L2 하이브리드

```
질문 → [L1: Jloot 라우터 51ns] → 禁 발견 → 즉시 차단 (RAG 호출 0원)
                                → 통과   → [L2: RAG 시맨틱 검색] → 답변
```

## 사용법

```bash
# 패킹: 폴더 → .jloot
jloot pack ./knowledge-base

# 인메모리 검색
jloot search knowledge.jloot "키워드"

# HTTP API 서버
jloot mount knowledge.jloot -p 7700
```

### API 엔드포인트

```
GET  /get?path=요한/3장/16절     → 텍스트 반환 (51ns)
GET  /search?q=사랑              → 전문 검색 (8.3ms)
GET  /ban?dir=출애굽기/20장/15절  → 거버넌스 체크
GET  /stats                      → 브레인 상태
POST /materialize                → 백그라운드 폴더 물질화
```

## 누구를 위한 것인가

|------|-------------|
| **군부대 / 국방** | 외부 통신 금지. USB에 .jloot 담아 배포 |

## 기술적 결정

### 왜 벡터DB가 아닌가

벡터DB는 "비슷한 의미"를 수학으로 찾습니다. 하지만:
- 임베딩 모델이 바뀌면 결과가 달라짐 (비결정론적)
- AI가 거짓 유사도를 따라 환각할 수 있음
- 서버/GPU/비용이 필요

Jloot은 **경로가 곧 주소**입니다:
- 같은 경로는 100% 같은 결과 (결정론적)
- `禁_` 파일이 있으면 AI가 물리적으로 차단됨
- Go 바이너리 1개면 어디서든 실행

### 왜 폴더인가

```
Redis:  SET "禁:도둑질" ""     ← 개발자만 가능
Jloot:  폴더 만들기 → 禁_도둑질  ← 세무사, 변호사, 누구나
```

**폴더를 만들 줄 아는 사람이면 AI를 통제할 수 있습니다.**
